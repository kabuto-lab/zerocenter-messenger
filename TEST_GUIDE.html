📡 ME55 Messenger — сборка и тестирование

Полная инструкция, как собрать бинарник, запустить два узла, проверить личные сообщения и групповой чат. Все команды для Windows / PowerShell.

Что должно быть готово: Rust toolchain (rustc 1.95+, cargo), VS Build Tools для линковщика MSVC. Если cargo build не запускается — у тебя нет toolchain'а.

1. Сборка

Headless CLI (тонкий бинарник ~9 МБ)

cargo build --release

Результат: target\release\ME55.exe. Этого достаточно для всех тестов в этом гайде.

Опционально — GUI (Tauri 2.x, ~5 минут на сборку)

cargo build --release --features gui

Запуск GUI: добавить флаг --gui к любой команде запуска.

2. Запуск двух узлов

Открой два терминала. В каждом запусти бинарник со своим профилем — профиль определяет, где хранится identity, БД, ключи в OS keyring.

Терминал 1 — Alice
.\target\release\ME55.exe --profile alice
Терминал 2 — Bob
.\target\release\ME55.exe --profile bob

При первом запуске ты увидишь баннер вида:

📡 ME55 Messenger
══════════════════════════════════════
Profile: alice
Peer ID: 12D3KooW...

Type 'help' for commands, 'quit' to exit.

>

Профили живут в %LOCALAPPDATA%\ME55\alice\ и ...\bob\ — там identity.json, messages.db, и т.д. DEK для шифрования at-rest лежит в Windows Credential Manager.

3. Тест личных сообщений (DM)

1 Узнай PeerId каждого узла:

whoami

2 Узлы на одной машине обнаружат друг друга через mDNS автоматически. Если этого не произошло — у Alice введи:

addr

Скопируй один из /ip4/127.0.0.1/tcp/<port>/p2p/<alice-pid> и у Bob введи:

connect /ip4/127.0.0.1/tcp/<port>/p2p/<alice-pid>

3 Проверь связь — у Bob:

peers

Должен увидеть Alice в списке.

4 Отправь сообщение — у Bob:

send <alice-pid> Привет, Alice!

У Alice в терминале появится:

🔓 <bob-pid>: Привет, Alice!
>

А у Bob:

📤 Encrypted message sent to <alice-pid>

5 Ответь от Alice (важно для следующего шага — групповой чат требует двусторонней DR-сессии):

send <bob-pid> Привет, Bob!

6 Проверь анти-MITM fingerprint (защита от подмены identity):

safety <peer-pid>

Сравни вывод в обоих терминалах — должны совпадать. Если совпадают — обмен ключей не был перехвачен.

4. Групповой чат

Prerequisite: между двумя узлами должна быть установлена двусторонняя DR-сессия. Если ты отправил DM в обе стороны (шаги 4 и 5 выше) — всё готово. Если только в одну — групповая раздача SenderKeyBundle упадёт в warn-and-skip.

1 Alice создаёт группу с Bob:

group create demo <bob-pid>

У Alice:

✅ Created group 'demo' id=<64-hex>
   Broadcast CreateGroup to 1/1 member(s)

У Bob (auto-join через DR-канал):

INFO  Joined group 'demo' (id=<gid>, founder=<alice-pid>)

2 Проверь у Bob, что группа появилась:

group list

Увидишь строку вида: <gid> 'demo' epoch=0 members=2 founder=<alice-pid>

3 Alice отправляет первое сообщение в группу:

group send <gid> Hello team from Alice

У Alice:

📨 Group message sent to 1/1 member(s) of <gid>

У Bob:

👥 [<gid-prefix>] <alice-pid>: Hello team from Alice
>

Под капотом на первой отправке: Alice генерирует SenderChain (HMAC-SHA256 + ChaCha20-Poly1305 + Ed25519), рассылает SenderKeyDistribution каждому участнику через 1:1 DR-сессии, потом шифрует текст один раз и фанаутит N-1 unicast'ом.

4 Bob отвечает:

group send <gid> Привет команде от Bob

У Alice в терминале появится: 👥 [<gid-prefix>] <bob-pid>: Привет команде от Bob

На этом моменте оба узла обменялись SenderChain bundles, каждый держит ReceiverChain для другого. Базовый сценарий group chat прошёл end-to-end.

5. Тест ротации ключей (forward secrecy на удалении)

Это проверка реализации forward secrecy: когда участник уходит, оставшиеся ротируют свои SenderChain так, чтобы кэшированные у ушедшего chain keys ничего не дешифровали впредь.

1 Bob выходит из группы:

group leave <gid>

У Bob:

✅ Left group <gid>; Leave broadcast to 1/1 member(s)

У Alice (обработка Leave + ротация):

INFO  Removed member <bob-pid> from group <gid> (Leave)
INFO  Rotated sender-chain for group <gid> at epoch 0;
      redistributed bundle to 0 member(s)

(redistributed = 0 потому что в группе осталась только Alice.)

2 Проверь, что у Bob группы больше нет:

group list

Должно быть пусто (или без demo).

Тест founder-only команд (опционально, нужен 3-й профиль)

Запусти третий узел с профилем carol:

.\target\release\ME55.exe --profile carol

У Alice (создавшей группу — она founder) — добавь Carol:

group add <gid> <carol-pid>

У Bob (не founder) попытка добавить кого-то даст:

❌ group add: only the founder can add members

У Alice — удалить участника:

group remove <gid> <carol-pid>

После этого Alice ротирует свою SenderChain (forward secrecy step) и оповещает оставшихся.

6. GUI режим (опционально)

Если собран с --features gui, добавь флаг --gui к запуску:

.\target\release\ME55.exe --profile alice --gui

Откроется Tauri-окно. Возможности:

  • Боковая панель с контактами и группами
  • Кнопка + New Group — создание группы
  • Для founder'а — кнопки + Add member, - Remove member, Leave
  • Push-refresh: входящее сообщение (DM или group) подгружается без перезахода в чат

8. Troubleshooting

deliver_kind: no session with <pid> (kind=1); group payload dropped

Между этими двумя узлами ещё нет 1:1 DR-сессии. Решение: отправь хотя бы один DM от того узла к этому (или наоборот) — после первого успешного обмена сессия зафиксируется в БД и групповая раздача пойдёт.

MembershipUpdate epoch N <= stored M — rejecting stale update

Это не ошибка — это защита от replay'а старых membership-апдейтов. Если бы атакующий скопировал старое "добавить X" и попытался применить его после "удалить X" — мы бы его отвергли.

Группа не появилась у второго узла после group create

Проверь:

  • Между узлами есть активное соединение? — peers у обоих
  • DR-сессия установлена? — попробуй send в обе стороны до group create
  • В трейсе у founder'а есть Broadcast CreateGroup to 1/1 member(s)? Если 0/1 — не доставлено.

Полный сброс состояния узла (если что-то застряло и хочется начать с нуля):

  1. Удалить директорию профиля: %LOCALAPPDATA%\ME55\<profile>\
  2. Удалить DEK из Credential Manager (опционально — иначе новый identity не сможет читать старую БД, но если её всё равно нет, это не важно)

Известный flaky-тест: storage::store::tests::mailbox_drops_basic_lifecycle иногда падает из-за race на границе секунды в chrono_time(). Перезапусти cargo test --lib — пройдёт. К продакшн-коду не относится.
