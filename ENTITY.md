## 2. Правила изменения кода

1. **Читать целиком.** Перед правкой файла — прочитать его полностью (`Read`), найти все вхождения символов по репо (`Grep`). Пути не выдумывать.
2. **Совместимость.** Не менять поведение без явной просьбы; правки точечные; стиль как у соседнего кода (4 пробела, `snake_case`, `Result<T>` через `anyhow`).
3. **Протокол и схема.** При изменении `ProtocolMessage`, `NetworkBehaviour`, `StreamProtocol` имён, или SQLite-схемы — обязательно указать версионирование и план миграции. Ломать сетевой протокол = ломать всех существующих пиров.
4. **Ключи и identity.** `identity.json` — чувствительный файл; не логировать приватные ключи, не сериализовать их в протокол, не пересылать по сети.
5. **UX надёжность.** Любая сетевая операция: успех / ошибка / таймаут / disconnect. CLI-команда должна сообщать пользователю, что произошло.
6. **Зависимости.** Не добавлять крейты без нужды. Перед `cargo add` — обоснование в обсуждении. Новая крипто-зависимость требует обоснования безопасности.
7. **Логи.** `tracing::info!` для событий уровня «соединение установлено», `debug!` для подробностей, `warn!`/`error!` для аномалий. `println!` — только для вывода, предназначенного пользователю CLI.
8. **Документация.** Не раздувать `README.md`/`DEV_CONTEXT.md` без запроса. История — короткие записи в §10 этого файла.

Формат работы ИИ: для крупных изменений — **контекст → затронутые файлы → что сделано → как проверить**; для мелочей — коротко. Для нетривиального (крипто, новый транспорт, изменение протокола, миграции БД) — режим **TLA Entity**, §9.

## 4. Фаза продукта

- **Phase 1 (P2P Foundation)** — ✅ готово (identity, swarm, mDNS, Kad-скелет, SQLite, CLI).
- **Phase 2 (Direct Messaging)** — ✅ готово (request-response DM, gossipsub-фоллбек, история).
- **Phase 3 (End-to-End Security)** — ⏳ в работе. Критическая цепочка: подпись отправителя → CBOR → Double Ratchet (X3DH + per-message keys) → удаление небезопасного gossipsub-фоллбека → replay-окно.
- **Phase 4+** — см. `project.html` → «Roadmap».

Ось приоритетов: **безопасность > корректность > функциональность > UX > производительность**. Ни одна новая фича не идёт вперёд незакрытых P0 из §6 `project.html`.

## 6. Релиз и распространение — без «двух ошибок»

Принцип: **один бинарь = один известный коммит = один известный протокол**. Несовместимые по протоколу пиры не должны молча «как будто подключаться».

Правила:

- **Версия протокола.** `StreamProtocol::new("/zerocenter/<feature>/X.Y.Z")` — при ломающих изменениях менять `X` или `Y`. Пиры со старым `X` отсекаются Identify-rejectom, а не зависают.
- **Identity vs profile.** `identity.json` привязан к профилю. Удаление профиля = новая идентичность = старые контакты вас не узнают. Не перезаписывать `identity.json` в процессе миграций.
- **SQLite-миграции.** Все `CREATE TABLE IF NOT EXISTS` идемпотентны. Новые колонки — через `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` с дефолтом. Перед релизом, меняющим схему — bump `PRAGMA user_version` и миграционный шаг.
- **Release-сборка.** `cargo build --release` с `lto = true`, `codegen-units = 1`, `strip = true` (уже в `Cargo.toml`). Перед выпуском — `cargo clippy -- -D warnings` и `cargo test`.
- **Windows-специфика.** `target\release\zerocenter.exe` — путь с обратными слэшами. Firewall должен пропустить бинарь, иначе mDNS/TCP не увидят соседей.
- **Проверка совместимости.** После любого изменения протокола/схемы — запустить старый бинарь и новый, убедиться, что они либо работают, либо корректно отказывают (не виснут).

Регламент выпуска:

| Когда                                  | Что сделать                                      |
|----------------------------------------|--------------------------------------------------|
| Закончил правки локально               | `cargo fmt`, `cargo clippy -D warnings`, `cargo test` |
| Готовлю билд                           | `cargo build --release`                          |
| Меняю протокол                         | bump version в `StreamProtocol` + запись в §10   |
| Меняю SQLite-схему                     | миграционный шаг + bump `user_version`            |
| Меняю идентичность/ключи               | документ о миграции профилей + предупреждение    |

Правило для ИИ при выполнении shell-команд: **строго одна команда за сообщение**, следующая — только после результата предыдущей (особенно важно для `cargo build`, который может идти минуты).

## 9. TLA Entity — Triple-Level Architect

**Когда включать:** новый модуль (voice, group chat, file transfer), изменение крипто-слоя (Double Ratchet, X3DH, замена Noise), новый транспорт (Obfs4, Snowflake, WebRTC), ломающее изменение `ProtocolMessage` или SQLite-схемы, релейный слой, любой код, касающийся приватных ключей. Тривиальные правки — без TLA.

**Триггер:** `TLA Entity, new task: [описание]` — ИИ входит в режим и не пропускает уровни.

**Обязательный ответ:** `Understood. Entering Three-Level Architect mode.`

### Уровни

**Level 1 — Strategic planning.**
Требования, threat model (для крипто — обязательно: что защищаем, от кого, какие гарантии — конфиденциальность, целостность, forward secrecy, post-compromise security), точки интеграции с существующими модулями, высокоуровневая архитектура. Кода нет.

**Level 2 — Architectural design.**
Структуры Rust (`struct`, `enum`, trait bounds), сигнатуры публичных функций, границы модулей (`src/crypto/ratchet.rs`, `src/protocol/envelope.rs` и т.п.), state machines (для хендшейков — диаграмма состояний), SQLite-схема (таблицы, индексы, `user_version`), wire format (CBOR-схема), изменения `NetworkBehaviour`. Бизнес-логику не писать.

**Level 3 — Incremental implementation.**
По одному файлу за шаг; перед следующим — согласование с человеком. Продакшен-качество: `Result<T>` везде, нет `unwrap()` в рантайме (кроме инварианта, комментарий обязателен), нет `panic!` в сетевом коде, корректная обработка `Drop`, аудит unsafe (у нас его быть не должно), идемпотентность storage-операций. Для крипто: константное время сравнений, zeroize секретов (`zeroize` крейт), тесты с известными векторами.

**Правила:**
- Только `src/*`, без создания параллельных деревьев.
- Стиль и паттерны как в текущем коде (`tracing`, `anyhow::Result`, `async fn` на tokio).
- Не ломать существующие пары ключей/БД без миграции.
- Для Level 3 подтверждение перед каждым файлом имеет приоритет над скоростью.
- Деньги/токены здесь не фигурируют (это messenger, не кошелёк) — но если появится криптоплатёж, суммы только через `u128`/`BigUint`, никогда `f64`.

## Подход ИИ-кодера (вытекает из документа)

Это не автопилот — это соавтор с дисциплиной:

- **Читает перед правкой.** Никогда не пишет в файл, не прочитав его.
- **Точечные изменения.** Не рефакторит соседний код без запроса, не добавляет зависимости зря.
- **Иерархия источников правды.** Перед сетевой правкой — `src/network/behaviour.rs` и `src/core/node.rs`. Перед крипто — план + threat model. Перед релизом — §6.
- **TLA для нетривиального.** Крипто, новый транспорт, ломающий протокол → три уровня (план → архитектура → один файл за шаг + согласование).
- **Без «памяти» между чатами.** Каждая сессия с нуля — `ENTITY.md`, `README.md`, `DEV_CONTEXT.md`, `project.html` заменяют память.
- **Безопасность выше скорости.** Регресс крипто или утечка ключей — стоп-краны, даже если просят «быстренько».
