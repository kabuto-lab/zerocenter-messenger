§1 — Executive Summary

ZeroCenter Messenger is a fully decentralised instant messenger: there is no server, no leader node, no operator and no directory. Every participant runs the same single binary (zerocenter.exe, ~9.3 MB), discovers peers over a Kademlia DHT plus mDNS, and exchanges messages directly over authenticated, encrypted libp2p streams. The threat model it targets is a Signal/Tor-class adversary: a network observer who can see, block and tamper with traffic, and who would like to learn who is talking to whom.

The project is mature for its age. In 40 commits over five weeks it has shipped five development phases: a libp2p transport, signed direct messaging, full Signal-style end-to-end encryption (Double Ratchet + X3DH), at-rest encryption keyed from the OS secret store, a pluggable wire- obfuscation transport that defeats DPI fingerprinting, a DHT store-and-forward mailbox for offline delivery, sealed-sender metadata protection, and Megolm-style group chats. It carries its own four-document audit pack (INVARIANTS.md, CRYPTO.md, THREAT_MODEL.md, SELF_AUDIT.md) and 119 unit tests.

Strengths

  • Disciplined cryptographic engineering. Every signature and KDF call carries an explicit domain-separator string; AEAD associated data binds sender/recipient identity; signature verification is ordered before state mutation in the ratchet and sender-chain code.
  • Defence in depth. Message bodies are encrypted on the wire and again at rest; the wire is optionally obfuscated against DPI; sender identity is sealed from network observers; the audit finding F12 (just landed) now even HMACs recipient PeerIds in the outbox.
  • Honest self-documentation. The audit/ pack enumerates 26 load-bearing invariants with file:line pointers and documents what the system explicitly does not protect.

Areas of attention

  • One 3,576-line file. core/node.rs concentrates the entire protocol state machine — the natural place for subtle bugs and the priority of the audit in §10.
  • No external review yet. The cryptography composes well-known primitives, but the integration is hand-rolled and has never been examined by a third party.
  • Metadata residue. Recipient PeerIds, slot identifiers and ACK records still leak who-talks-to-whom to a DHT observer; this is documented, not yet closed.

§2 — System Architecture

The node is a single async tokio process. A foreground UI thread (Tauri webview or a line-reader CLI) feeds NodeCommands over an mpsc channel into a background swarm task that owns all networking and cryptographic state. The layers, top (application) to bottom (wire):

UI  Tauri 2 webview (gui/app.rs, dist/index.html) — default; or headless CLI REPL (cli.rs). Both emit NodeCommand over mpsc.
↓
Node / Orchestration  core/node.rs — the swarm event loop, DM/group send-receive-bootstrap state machine, prekey & OTPK lifecycle, mailbox scheduler.
↓
Protocol & Crypto  protocol/ wire envelopes · crypto/ Double Ratchet, X3DH, Megolm sender chains, sealed sender, at-rest DEK.
↓
Storage  storage/store.rs — SQLite (messages.db), every sensitive blob AEAD-wrapped under the per-profile DEK.
↓
Network behaviour  network/behaviour.rs — libp2p Kademlia + gossipsub + mDNS + identify + two request-response protocols · network/mailbox.rs DHT store-and-forward.
↓
Transport  TCP → network/scramble.rs ScrambleStream obfuscation (optional) → Noise (auth+encrypt) → Yamux (mux). QUIC disabled on this toolchain.

Two design decisions are load-bearing and worth calling out:

  • DMs are request-response, not gossipsub. A gossipsub fallback was deliberately removed to avoid broadcasting (even encrypted) direct messages to the whole mesh. DMs require a live connection or the DHT mailbox; gossipsub now carries only a vestigial global topic.
  • The transport stack is custom. The ScrambleStream is spliced in via SwarmBuilder::with_other_transport, not the stock with_tcp, so the obfuscation handshake runs on the raw socket before the Noise upgrade.

§6 — Core Data Flows

6.1 First DM to a stranger (X3DH bootstrap)

  1. User runs send <peer> <text>. No ratchet session exists.
  2. Node fires a PrekeyRequest on /zerocenter/prekey/1.0.0; the plaintext is buffered in pending_sends.
  3. Responder returns a signed X25519 prekey, optionally with a one-time prekey (OTPK).
  4. Initiator verifies the Ed25519 signature against the responder's PeerId, runs X3DH-lite (3-DH if an OTPK was supplied, else 2-DH), derives the root key, and builds an initiator RatchetState.
  5. The buffered message is ratchet-encrypted, wrapped in a signed (or sealed) ProtocolMessage, and sent — or queued to the outbox + DHT mailbox if the peer is offline.

6.2 Receiving a DM

  1. process_incoming_dm rejects expired envelopes first, then verifies the Ed25519 signature.
  2. For non-sealed envelopes it cross-checks the libp2p transport peer against the signed sender; for sealed envelopes that check is deliberately skipped (the transport peer is decoupled by design).
  3. If no session exists, the first-message header (X3DH ephemeral + OTPK id) drives a responder X3DH bootstrap; the new session is committed to memory only after the first decrypt succeeds.
  4. Plaintext is stored AEAD-encrypted; a GUI DmReceived event fires; for mailbox-fetched messages an ACK record is published to the DHT.

6.3 Group message (Megolm fan-out)

  1. Sender loads or creates its own SenderChain for the group; if fresh, the sender-key bundle is distributed to every member over their 1:1 ratchet sessions.
  2. The message is Megolm-encrypted once (chain key advance + ChaCha20-Poly1305 + per-message Ed25519 signature).
  3. The single ciphertext is fanned out as N−1 unicast kind=2 envelopes — confidentiality from Megolm, transport security from each 1:1 session.
  4. On remove/leave, every remaining member rotates its SenderChain so a departed peer's cached key is dead for future messages (forward secrecy, best-effort, not retroactive).

6.4 Offline delivery (DHT mailbox)

  1. When a peer is unreachable, the already-encrypted wire bytes go to both the on-disk outbox and a Kademlia record at drop_kad_key(recipient, sender, slot); the sender also start_providings the slot key.
  2. Recipients poll get_providers for recent hourly slots, fetch each drop, and route it through the normal receive pipeline.
  3. On successful decrypt the recipient publishes an ACK record; the sender stops republishing that drop (else it republishes until a 7-day cap).

The P2PNode struct holds all live state in memory. Several maps are keyed by remote PeerId and are therefore attacker-influenceable — a deliberate focus of the audit:

§10.2 — High-severity findings

H1HighStale/replayed first message can overwrite a live ratchet session
Location
core/node.rs:1830-1872 (process_incoming_dm step 5), :3219-3241 (process_pending)
Issue
The receive path chooses "run a fresh X3DH bootstrap" vs "use the existing session" purely on the absence of a stored session. There is no first-message idempotency guard on the 2-DH path (the OTPK consumed_at gate only covers 3-DH), and decrypt_first_message inserts the new session before attempting decrypt.
Impact
An active attacker who replays the captured first ProtocolMessage of a conversation — while the live session row is momentarily unavailable — re-derives the same SK, builds a fresh responder state at index 0, decrypts the genuine old message successfully, and persists it over the advanced live session. Every subsequent real message then fails AEAD; the conversation is permanently desynced until a new handshake. The audit-F9 guard only protects the decrypt-failure case.
Fix
Gate responder bootstrap on the header being pn==0 && n==0; refuse to bootstrap when a session blob exists but failed to load; for 2-DH first messages add a replay marker (record the first-message ephemeral, reject a second bootstrap with the same one).
H2Highrestore_session_if_persisted fails open on DB error / corrupt blob
Location
core/node.rs:3439-3464
Issue
The function returns false for three distinct conditions — no row, a corrupt/undeserialisable blob, and a DB or DEK-decrypt error. Every caller treats false as "no session — safe to bootstrap fresh".
Impact
A transient SQLite error or a DEK-decrypt failure on a real, live session causes the node to silently abandon the ratchet and bootstrap a new one. This is fail-open behaviour and the enabling primitive for H1. INVARIANTS §10 requires at-rest decrypt failures to propagate, not be swallowed into a boolean.
Fix
Return a three-state result (Present / Absent / Error). On Error or "corrupt blob present", callers must drop the message / abort the send — never pave over the session.
H3Highpending_recvs/pending_sends unbounded → memory-exhaustion DoS
Location
core/node.rs:44-48 (fields), :1041, :1856-1859
Issue
Both maps are keyed by remote PeerId with no cap on key count, no cap on payloads per peer, no byte cap and no TTL. Entries are drained only if a valid prekey response later arrives; the hourly cleanup tick never touches them.
Impact
A Sybil attacker mints unlimited valid PeerIds, sends one validly-signed first-message envelope per identity (each carrying a large ct) and never answers the prekey fetch. pending_recvs grows without bound until OOM — an unauthenticated-cost remote crash.
Fix
Cap the peer-key count (LRU-evict), cap payloads per peer, cap total bytes, add TTL eviction on the cleanup tick; also bound inflight_prekey_fetches. Note: this is the project's own deferred finding F4 — rated there "Low, noise only"; this audit re-rates it High.
H4HighStored XSS — remote message content reaches webview innerHTML unescaped
Location
dist/index.html:655-663 (renderMessages)
Issue
Every other render path escapes user-controlled strings via escapeHtml(); renderMessages interpolates msg.sender and msg.content straight into innerHTML. msg.content is the decrypted plaintext of a message from a remote peer.
Impact
A remote peer sends a DM or group message whose plaintext is <img src=x onerror=...>. When the victim opens that conversation the markup parses and the script runs inside the Tauri webview — where window.__TAURI__.core.invoke is in scope. The injected script can call every exposed Tauri command (send_message, add_contact, create_group, leave_group, …): full webview compromise / account takeover, triggered purely by messaging the victim.
Fix
escapeHtml() both fields (or build the bubble with textContent/createElement). Add a restrictive Content-Security-Policy (default-src 'self'; script-src 'self') in the Tauri config.
H5HighUnbounded inbound message deserialization → allocation / OOM DoS
Location
protocol/message.rs (ProtocolMessage/EncryptedPayload::from_bytes), protocol/group.rs (GroupControl), network/behaviour.rs:139-159
Issue
All wire parsing runs serde_json::from_slice on attacker-controlled bytes with no size cap before deserialization; the request-response protocols use Config::default() with no explicit request-size maximum. Signature/expiry verification happens after from_bytes. GroupControl::members is Vec<Vec<u8>>, amplifying parse cost.
Impact
A peer that can deliver bytes (direct DM, DHT-mailbox drop, or group control) sends a multi-megabyte/gigabyte document; the receiver allocates it all — plus two further clones of ct — before any check runs. Memory pressure / OOM on the single tokio task that also runs all crypto: an event-loop-starving DoS that amplifies H3.
Fix
Set a conservative request-size maximum on both request-response protocols; reject any envelope exceeding a few hundred KiB before deserialization; cap members/added/removed lengths after parse.

§10.3 — Medium-severity findings

M1Mediumprocess_pending re-bootstraps a session per queued payload
Location
core/node.rs:3228-3241
Issue
When a prekey response arrives, process_pending iterates every queued payload and may call bootstrap_responder_and_decrypt for each. Bootstrap is not idempotent — on a decrypt failure it removes the session without persisting, so the next payload bootstraps again. Queued payloads are in arrival order, not chain order.
Impact
A peer who pipelines several first-message-shaped envelopes before the prekey lands causes repeated session install/remove cycles and can land the session in a wrong state. Shares a root cause with H1.
Fix
Bootstrap the responder session exactly once per peer (identify the genuine first message by pn==0 && n==0), persist it, then decrypt the remaining payloads against it.
M2MediumMembershipUpdate accepts any forward epoch jump → group membership freezes forever
Location
core/node.rs:2150-2211
Issue
The MembershipUpdate verifier enforces epoch > stored (monotonicity) but not epoch == stored + 1 (continuity). Any strictly-greater epoch is accepted and the local epoch jumps to it.
Impact
One founder MembershipUpdate (malicious or buggy) with epoch = stored + 1000 — or near u64::MAX — is accepted; every legitimately-issued later update then has a smaller epoch and is permanently rejected as stale. Membership management for that group is bricked on the receiving node.
Fix
Require epoch == stored + 1; reject both stale (<=) and gapped (>) updates. If offline catch-up needs gaps, cap the forward jump and buffer-and-order.
M3MediumMembershipUpdate can remove the founder → desyncs all receivers
Location
core/node.rs:2150-2229
Issue
The MembershipUpdate arm applies the removed set without checking that it does not contain the group's own founder_pid.
Impact
A founder MembershipUpdate listing the founder in removed drops the founder from every receiver's group_members table while groups.founder_pid is unchanged. process_group_message then rejects the founder's own group messages and sender-key distributions as "from a non-member" — the group is desynced for everyone, recoverable only by re-creating it.
Fix
Reject (or filter) a MembershipUpdate whose removed set contains the founder PID — the founder cannot be removed by a membership update.
M4MediumUnauthenticated mailbox ACK lets any observer kill an offline delivery
Location
core/node.rs:1682-1738, network/mailbox.rs:121-128
Issue
Mailbox ACK records are unauthenticated (acknowledged in INVARIANTS §21). The sender's republish loop retires a drop on any FoundRecord at ack_kad_key without validating it. ack_kad_key = SHA-256(domain ‖ recipient ‖ sender ‖ slot) — all values an on-path observer already sees.
Impact
Any third party who can compute the ACK key publishes a forged empty record there; the legitimate sender permanently stops republishing a real, undelivered drop. Once Kad's per-record TTL lapses the genuinely-offline recipient can never retrieve it — a silently-dropped end-to-end message.
Fix
Have the recipient Ed25519-sign the ACK (over domain ‖ drop_kad_key) and verify it before mailbox_drop_ack. At minimum, let an ACK only reduce republish frequency, never retire the local drop row.
M5MediumKad query-tracking maps leak on the Found* result arms
Location
core/node.rs:60-71 (fields), :1582/:1610/:1626
Issue
pending_provider_queries and pending_record_queries are removed only on the Err and FinishedWithNoAdditionalRecord arms; the FoundProviders/FoundRecord arms dispatch their work but never remove the QueryId. A query completing without the expected terminal variant leaks its entry.
Impact
Slow unbounded growth of two maps over process lifetime, proportional to mailbox poll/republish cycles and partly attacker-amplifiable. No GC path.
Fix
Remove the QueryId on the Found* arms once per-result work is dispatched, or age-sweep all three maps on the cleanup tick.
M6MediumRatchet AEAD doesn't structurally bind peer IDs
Location
crypto/ratchet.rs:421-464
Issue
INVARIANTS §6 requires the ratchet AEAD AD to be ratchet_ad(sender, recipient) ‖ header. ratchet.rs only appends the header; the peer-ID portion is whatever the caller passes. The crypto layer cannot enforce the invariant — it delegates it to every node.rs call site.
Impact
Current callers do pass ratchet_ad, so no live exploit. But any future path calling encrypt/decrypt with empty or inconsistent AD silently loses the cross-peer/cross-session binding §6 promises, with no compile-time or runtime error.
Fix
Make encrypt/decrypt take explicit sender/recipient parameters and build the length-prefixed peer-ID prefix internally, so the invariant is structurally impossible to violate.
M7MediumRatchetState root/chain keys not zeroized on drop
Location
crypto/ratchet.rs:126-145, :394-415
Issue
RatchetState holds rk (root key), cks, ckr as bare [u8;32] and neither derives ZeroizeOnDrop nor impls Drop; it is also Clone. kdf_ck returns owned message keys dropped after the AEAD call without zeroization. SkippedKey is ZeroizeOnDrop — the discipline is applied inconsistently.
Impact
A process memory dump (core dump, swap, debugger) recovers the root and chain keys of every live session. With rk plus the peer's next DH pubkey an attacker derives all subsequent message keys — total loss of forward secrecy from the dump point. CRYPTO.md §14 claims zeroize is used for exactly this.
Fix
Make RatchetState ZeroizeOnDrop; wrap rk/cks/ckr in Zeroizing; zeroize message keys after the AEAD call; scrutinise the Clone derive.
M8MediumDEK left in unzeroized String/Vec copies after load
Location
crypto/keyring.rs:49-68
Issue
load_or_create_dek reads the DEK as a hex String from the keyring, hex-decodes it into a Vec<u8>, copies into [u8;32], and returns — none of the intermediates (nor the hex string in the create path) is zeroized.
Impact
The master key for all at-rest encryption leaves up to three unzeroized copies (raw + hex) in process memory. A memory dump or swap leak then compromises every at-rest blob — ratchet sessions, OTPK privates, stored plaintext, outbox.
Fix
Use Zeroizing<String>/Zeroizing<Vec<u8>> for every intermediate; zeroize the hex string after set_password; consider returning Zeroizing<[u8;32]>.
M9MediumScrambleStream read path doesn't bound the descrambled frame length
Location
network/scramble.rs:310-355
Issue
The descrambled 16-bit frame-length header is fully attacker/keystream-determined. The read path trusts payload_len up to 65535 with no check against MAX_PAYLOAD_PER_FRAME (1022) — the {256,512,768,1024} frame-size invariant (INVARIANTS §17) is enforced only on the write side.
Impact
A malformed/hostile peer at the obfs layer makes the receiver perform large reads and keystream work on garbage per bogus frame. Not catastrophic (Noise above still fails to authenticate) but a wasted-work amplification vector and a latent correctness gap.
Fix
On the NeedHeader→InBody transition, reject payload_len > MAX_PAYLOAD_PER_FRAME with InvalidData.
M10MediumScrambleStream::new is pub and builds a two-time-pad stream
Location
network/scramble.rs:226-255
Issue
ScrambleStream::new is pub and constructs the stream with identical (key, nonce) for both directions — a two-time-pad. The doc says "unidirectional tests only" but nothing enforces it; the production path correctly uses derive_ntor_keystream_pairs.
Impact
A future caller wiring ScrambleStream::new into a real bidirectional connection silently reintroduces audit finding F1 (XOR of the two directions reveals XOR of plaintexts). The compiler will not catch it.
Fix
Make ScrambleStream::new #[cfg(test)]-only or pub(crate)+#[doc(hidden)], or derive two distinct nonces so even the convenience path can't two-time-pad.
M11MediumAt-rest AEAD uses empty AAD → disk attacker can graft/rollback blobs
Location
storage/store.rs:417-462
Issue
encrypt_at_rest passes empty AAD (documented in INVARIANTS §11). The version byte and nonce don't bind row identity (peer_id, group_id, rowid) either.
Impact
An attacker with write access to messages.db can swap blobs between rows — move a ratchet state_blob into another peer's row, or replay an old state_blob to roll a ratchet backward — and the AEAD tag still verifies (keyed only by the DEK). The OTPK replay gate and ratchet rollback resistance partly rest on DB integrity the AEAD doesn't provide. Outside the stated v1 threat model, but worth closing.
Fix
Bind the row's stable key into the AAD (aad = table_tag ‖ peer_id/group_id); bump AT_REST_VERSION to 2 with a v1-read path.

§10.6 — Strengths confirmed by the review

  • Domain separation is thorough and complete. Every distinct KDF/signature context has its own separator; the cross-protocol replay class (INVARIANTS §1) is closed and actively test-pinned.
  • The §2 transport-peer↔signed-sender cross-check is correct — correctly skipped for sealed envelopes, and correctly ordered (verify → cross-check → expiry → decrypt).
  • Megolm verify-before-mutate is correct — the Ed25519 signature is checked before any skipped-cache or chain mutation; a forged signature is a no-op on chain state.
  • The OTPK three-state lifecycle (served_at vs consumed_at) is carefully reasoned; atomic UPDATE … RETURNING gives single-use and the F3 replay gate holds.
  • SQL is uniformly parameterised — no string-interpolated queries anywhere in the storage layer.
  • Wire parsers are bounds-checked — pop_bytes, extract_inline_pubkey, group_row_from_db and the Megolm signature try_into all reject malformed input instead of panicking.
  • The ScrambleStream F1 fix is real — derive_ntor_keystream_pairs derives genuinely independent per-direction keystreams; generate_representable_keypair has a hard 64-attempt cap.
  • At-rest encryption is genuinely verified — the *_is_encrypted_at_rest tests scan raw on-disk bytes; wrong_dek_fails_to_decrypt confirms the failure path.
  • No plaintext in info!/warn! logs — message plaintext is consistently at debug! (INVARIANTS §19 holds).
  • Bounded skipped-key caches — both the ratchet and Megolm cap n_to_skip + skipped.len() before the derive loop with saturating arithmetic.
  • The keyring is fail-loud — an unreachable keyring warns and uses an ephemeral DEK rather than silently dropping to plaintext-at-rest.
  • escapeHtml is correct and applied in every render path except the one flagged in H4 — which is exactly why H4 is a real bug rather than a systemic gap.

§11 — Conclusions & Remediation Priority

The audit found no Critical issues and confirmed the cryptographic foundations: there is no nonce/IV reuse, no missing domain separator, no signature-verification bypass, and no key reuse in the handshake or ratchet. The primitives and their composition are sound. The 5 High and 11 Medium findings cluster into three themes:

  • Session-state integrity (H1, H2, M1). A single root cause — the receive path conflates "no session", "DB error" and "corrupt blob" into one boolean, and responder bootstrap is not idempotent against a successfully-decrypting stale first message. This is the most important cluster: in the field it would most likely be mis-diagnosed as a "flaky session" bug.
  • Resource exhaustion (H3, H5, M5, L4). Several unauthenticated-cost remote DoS vectors — unbounded pending_* maps, unbounded inbound message size, leaking Kad query maps, and per-message X25519 on the single event-loop thread.
  • Webview compromise (H4). A remotely-triggerable stored XSS that, because the webview holds the Tauri invoke bridge, escalates to full command execution.

The Medium tier is otherwise dominated by group-chat control-plane gaps (M2, M3) and memory-hygiene / zeroization gaps (M6–M8) — consistent with the project's own note that the group-chat track and its invariants §24–§26 post-date the last self-audit. Notably, this review re-rates the project's own deferred finding F4 (unbounded pending_recvs) from "Low — noise only" to High.

Recommended remediation order. (1) H4 and H5 first — remotely triggerable, small well-scoped fixes. (2) H1 / H2 / M1 as one coordinated change to the session-restore logic (three-state result + idempotent bootstrap). (3) H3 and the resource-exhaustion tier (M5, L4) — bounds, caps and TTL eviction. (4) The zeroization cluster (M7, M8, L7, I2) — mechanical, low-risk. (5) The group-chat control-plane fixes (M2, M3, L2) and the remaining Low/Informational items. None of the findings breaks the core end-to-end-encryption guarantee, but all five High findings should be closed before any external audit or release.
