ME55 — Web Edition Development & Integration Plan · 2026-05-20

A browser-native client for ME55 Messenger that interoperates with the existing Rust/libp2p nodes. Companion to ROADMAP.html — same project, new surface.
Status: Planning — nothing built yet. This document is the proposal: architecture decision, what is reused, a phased build order, and the security deltas a browser surface introduces.

1 · Goal & the browser reality

The native client is a P2P node: raw-TCP libp2p, mDNS LAN discovery, a Kademlia DHT mailbox, SQLite at rest, and the long-term key in the OS keyring. A browser tab can do none of those directly. Before any plan, the hard constraints:

Native capabilityIn a browserWeb-edition answer
Raw TCP + Noise + Yamux transportimpossible WebSocket (WSS) to a relay; WebRTC/WebTransport peer links later.
mDNS LAN peer discoveryimpossible None — a web client always reaches the network through a relay.
Inbound dial (be a listener)impossible Browser only dials out; the relay is its always-on delivery agent.
SQLite (rusqlite)no IndexedDB (or OPFS) behind a storage trait.
OS keyring DEKno Passphrase → Argon2id → key-wrap; unlocked per session.
Always-on background processthrottled Tabs are frozen/killed — rely on the DHT mailbox + future Web Push.
Double Ratchet / X3DH / Megolm cryptoyes — WASM This is the asset. The crypto crate is pure Rust; it compiles to WASM unchanged.

2 · The non-negotiable principle

The web client does its own end-to-end encryption. The server never sees plaintext or long-term keys. ME55 exists to be zero-trust; a web version that ships crypto to a server is a different (worse) product. Therefore the browser runs the same Double Ratchet code as the native client — compiled to WASM, not reimplemented in JavaScript. One crypto implementation, one audit surface. (Reimplementing in JS would mean a second copy to keep in sync — and the OTPK serve/consume bug fixed on 2026-05-20 is exactly the class of drift that causes.)

4 · Chosen architecture (Option B)

  |  Web UI  (ported dist/      |              |  WebSocket listener    |        | native nodes   |
  |          index.html)        |   WSS        |  (libp2p-websocket)    |  TCP   | DHT mailbox    |
  |    |                        | <==========> |        |               | <====> | other relays   |

  |    Megolm / sealed sender   |  only        |  - Kademlia mailbox    |   relay NEVER decrypts:
  |    |                        |              |  - identify / kad      |   it forwards & polls

Browser side — me55-web

Three layers. me55-core (WASM): the crypto + protocol crate, wasm-bindgen-exported — identity, X3DH, ratchet encrypt/decrypt, Megolm, sealed-sender wrap. Storage adapter: implements the Store trait against IndexedDB; same AEAD-at-rest as native (the DEK just comes from the vault, not a keyring). Key vault: on first run generates the Ed25519 identity + DEK, wraps them with an Argon2id-derived key from the user passphrase, persists the wrapped blob; unlock decrypts into WASM memory for the session.

Relay side — new --relay role of the existing binary

Not a new program — a role flag on me55-node. It runs the normal libp2p swarm plus a WSS listener. For each connected browser it (1) forwards the browser's outbound sealed envelopes onto the request-response / mailbox paths, and (2) polls that web identity's DHT mailbox slot and pushes drops down the socket. It authenticates the socket with a sign-a-nonce challenge against the claimed identity key, but it handles only ciphertext — it is a delivery agent, not a trusted party.

6 · Prerequisite — workspace split

The repo is a single crate today. Web work starts by splitting it into a Cargo workspace so the crypto serves both targets from one source:

me55/
  Cargo.toml            # [workspace]
  me55-core/            # crypto + protocol + Store trait + Identity
                        # no tokio / libp2p / rusqlite / keyring
                        # builds for native AND wasm32-unknown-unknown
  me55-node/            # the native binary: libp2p swarm, CLI, --relay role,
                        # rusqlite Store impl, OS keyring
  me55-web/             # wasm-bindgen crate: exports me55-core to JS,
                        # IndexedDB Store impl, key vault
  web/                  # the browser UI (adapted from dist/index.html)

Gotchas to settle during the split: getrandom needs its js feature for OsRng under wasm; chrono needs wasmbind for Utc::now(); the crypto must stay sync (no tokio) so it is callable from WASM.

7 · Phased delivery plan

  • W0 — De-risk spike. ~2 days
    Compile the crypto module to wasm32-unknown-unknown, confirm OsRng works in-browser, run one Double Ratchet round-trip under wasm-pack test --headless. Done when: a headless-browser test does an X3DH + encrypt + decrypt and passes.
  • W1 — Workspace split.
    Carve out me55-core / me55-node / me55-web; extract the Store trait. Done when: native build + 117 tests still green, and me55-core builds for the wasm target.
  • W2 — Browser storage & key vault.
    IndexedDB Store impl with the same AEAD-at-rest; passphrase → Argon2id → unwrap identity + DEK. Done when: identity survives a tab reload; rows in IndexedDB are ciphertext.
  • W3 — Relay role.
    me55-node --relay: WSS listener, libp2p-websocket transport, browser↔relay envelope framing, sign-a-nonce auth. Done when: a browser socket can hand a sealed envelope to the relay and it appears on libp2p; relay logs prove no decrypt.
  • W4 — Web DM send & receive.
    WASM core drives X3DH + ratchet; outbound sealed envelopes via the relay; inbound via relay push + mailbox poll; encrypt-once parity. Done when: a browser tab and a native node DM each other and their safety numbers match.
  • W5 — Web UI.
    Adapt dist/index.html to call the WASM core + socket instead of Tauri invoke(). Done when: contacts, DM history, and safety-number verification work in a plain browser tab.
  • W6 — Group chats in the browser.
    Megolm sender/receiver chains in WASM; group fan-out routed through the relay. Done when: a web user creates/joins a group with native members and messages flow both ways.
  • W7+ — Toward Option C (future).
    WebRTC browser-to-browser links, WebTransport, a PWA shell with Web Push for background delivery, multi-relay failover. Crypto and storage layers are untouched by this — only the transport hop changes.

    • 8 · Integration with the existing messenger

    Interop is free — if the wire format is shared

    Because me55-core is the same crypto the native node already runs, a web user and a native user share X3DH, the ratchet, Megolm, and the safety-number hash. A cross-platform conversation needs no bridge — it is the same protocol on both ends.

    Concrete finding — web clients must use sealed sender. process_incoming_dm enforces transport-peer == signed-sender for direct envelopes. A relayed web user's transport peer is the relay, not the user — so a plain envelope would be rejected. Sealed envelopes deliberately skip that check (the signature inside the seal authenticates the sender). So: all web-client outbound traffic is sealed-sender, and the relay/mailbox paths already support exactly that. This is a constraint, not new work.

    Delivery to a web user

    A native sender treats an offline web user like any offline peer: it writes a drop to the DHT mailbox. The web user's relay polls that mailbox slot and pushes drops down the socket. The relay is the web user's always-on mailbox poller — holding only ciphertext.

    Protocol versioning

    The web client targets the current /ME55/<proto>/<ver> strings unchanged. Any envelope-field change must land in me55-core and ship to both surfaces together — the workspace split makes that automatic, a JS reimplementation would not.

    9 · Security deltas a browser surface introduces

    ConcernWhy it is newMitigation
    Relay sees metadata The relay observes who connects, when, envelope sizes, and recipient PeerIds. Mandatory sealed sender (hides the sender); size padding; support multiple/self-hosted relays; document the relay as a metadata-trust boundary in the audit pack.
    Browser key storage < OS keyring IndexedDB is readable by any script on the origin; an XSS = identity-key theft. Strict CSP, no third-party scripts, Subresource Integrity, passphrase-wrapped keys at rest, optional WebAuthn/passkey-gated unlock, short in-memory key lifetime.
    No background delivery A closed/frozen tab receives nothing live. DHT mailbox covers offline; add a Service Worker + Web Push (W7) for wake-ups.
    No mDNS / no LAN-only mode A web client cannot run relay-free. Accepted; native client remains the relay-free option. Allow user-supplied relay URLs.
    Audit scope grows WASM build, key vault, IndexedDB store, relay are all new. Crypto stays the single audited me55-core; the audit pack gains a web addendum for vault + relay + CSP. The external audit (open roadmap item) should follow, not precede, W4.

    10 · Risks & open questions

  • WASM toolchain friction. getrandom/js, chrono/wasmbind, and keeping the crypto path tokio-free are known, solvable, but must be proven in W0 before committing to the plan.
  • Relay availability & DoS. The relay is a new always-on dependency and a new attack target. Needs connection rate-limits, per-identity quotas, and a story for running several.
  • libp2p-in-WASM maturity (Option C). webtransport/webrtc transports for rust-libp2p are evolving; deferring C past v1 keeps that risk out of the critical path.
  • IndexedDB quota & eviction. Browsers may evict storage under pressure; needs a persistence request and a "your history may be cleared" UX honesty note.
  • Passphrase UX. Lose the passphrase = lose the identity (no keyring, no recovery). Needs an explicit export/backup flow for the wrapped key blob.

    • 11 · Tooling & verification

    New to the toolbox: wasm-pack / wasm-bindgen, wasm-pack test --headless for crypto tests under a real browser engine, and a static host for the web/ bundle. Existing gates stay: cargo test --lib for me55-core + me55-node, cargo build --release for the native binary. A web↔native DM and a mixed-membership group message become the two new smoke tests.

    Source of truth for the native roadmap: plans/ROADMAP.md / ROADMAP.html. This plan is a proposal — architecture (Option B), phase order, and the sealed-sender constraint are the decisions to confirm before W0.
