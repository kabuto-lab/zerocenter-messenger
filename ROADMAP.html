ME55 Messenger — Roadmap updated 2026-05-24

Censorship-resistant, zero-trust, leaderless P2P messenger. Rust + libp2p 0.53. Source of truth is plans/ROADMAP.md; this page is a visual snapshot of it. Companion docs: ABOUT.html (plain-language overview), ANALYSIS.html (architecture deep-dive + 35-finding audit), RELAY.md / BEGET_SETUP.md (operator guides for Phase 6).
Phases 1 → 5 implemented and test-green. In flight: Phase 6 (bootstrap + relay for NAT traversal), Phase 7 (post-quantum hybrid X3DH via ML-KEM-768), opt-in deniable DMs. Outstanding: external security audit (needs a third party), public bootstrap nodes (Beget VPS pending). cargo test --lib last green at 119 / 119 on commit f5c2e65; in-flight work adds ~26 test functions on disk that have not been re-verified yet (clean build pending).

Phase 6 — Bootstrap + Relay (NAT traversal) In flight

Code on disk (uncommitted as of 2026-05-24, ~2500 LOC) but builds clean and the local 3-process smoke (relay + Alice + Bob, all on 127.0.0.1) works end-to-end. Public-VPS validation is pending — see BEGET_SETUP.md for the operator runbook.
ItemStatusNotes
libp2p dcutr feature + Circuit Relay v2in flight Two NAT'd peers connect through a public relay; DCUtR then attempts hole-punching to upgrade to a direct connection. Relayed bytes use plain Noise (we can't scramble a third party's circuit); direct upgraded bytes use the ScrambleStream wrapper.
CLI: --relay, --relay-server, --daemon, --bootstrap, --no-default-bootstrap in flight NAT'd clients pass --relay to listen on a circuit-relay address; VPS nodes set --relay-server + --daemon for systemd-friendly headless operation.
Dynamic bootstrap discovery (src/network/bootstrap.rs)in flight 4-layer resolution: CLI override → local PEX cache (peer_cache SQLite) → signed HTTPS manifest at bootstrap.me55.network/manifest.json (Ed25519 detached sig pinned to MAINTAINER_PUBKEY) → DNSADDR TXT fallback → hardcoded DEFAULT_BOOTSTRAPS. Maintainer pubkey is currently a 32-byte zero placeholder.
New binary: ME55AGUIin flight GUI-defaulting twin of ME55.exe. Sets #![windows_subsystem = "windows"] so double-clicking the exe opens only the Tauri webview — no extra console window. required-features = ["gui"] in Cargo.toml. Common startup body lives in src/entry.rs.
New binary: me55-toolsin flight Maintainer CLI: generate the Ed25519 keypair that signs the bootstrap manifest, print the pubkey to paste into MAINTAINER_PUBKEY, sign a manifest JSON. Headless; ships with the cargo build --release default set.
Beget VPS as bootstrap-1queued Operator runbook at BEGET_SETUP.md. End-to-end: ufw / systemd unit (with NoNewPrivileges, ProtectSystem=strict, capability set zeroed) → first-boot PeerId capture → multiaddr baked into DEFAULT_BOOTSTRAPS → rebuild + ship.
Bootstrap nodes 2 & 3 (cross-jurisdiction)queued Oracle Cloud Free in US-East + EU-Frankfurt. Beget is RU; we want at least two non-RU bootstraps before announcing the network, to keep a compelled-logs scenario from being a single-jurisdiction risk. THREAT_MODEL.md §Beget reminder.

Phase 7 — Post-quantum X3DH (PQXDH-style) In flight

ItemStatusNotes
ML-KEM-768 (FIPS 203) hybrid in X3DH first messagein flight ml-kem = "0.2" dependency wired. Long-term ML-KEM keypair lives on the identity record alongside the Ed25519 / X25519 keys. First message carries the 1088-byte ciphertext; hybrid SK = HKDF(X25519 outputs ‖ ML-KEM ss). Secure as long as either primitive holds. Matches Signal PQXDH (2023) and iMessage PQ3 (2024).
Downgrade-attack mitigationin flight Distinct HKDF info tags ME55-x3dh-pq-v1 / ME55-x3dh-otpk-pq-v1 so a peer stripping the PQ ciphertext from a captured envelope cannot recover a classical-X3DH secret that matches a hybrid initiator.
PQ signed-prekey publicationin flight Responder publishes its ML-KEM encapsulation key bound by an Ed25519 signature over a distinct domain (ML_KEM_PREKEY_SIG_DOMAIN). Initiator verifies before encapsulating.
Migration path for pre-PQ identitiesin flight Backward-compatible serde defaults — pre-PQ identities upgrade in place by generating an ML-KEM keypair on first load and republishing the prekey bundle. Old sessions continue on classical X3DH until the next session reset.

Opt-in deniable DMs In flight

ItemStatusNotes
--deniable-dm CLI flagin flight Outgoing 1:1 DMs sent WITHOUT the per-message Ed25519 signature. Authenticity rests on the ratchet AEAD MAC (shared between the two session participants → either could have forged it → neither can prove authorship to a third party).
Wire incompatibility notecaveat Peers expecting the legacy signed path drop deniable envelopes. Opt-in only; default remains non-deniable for v1. Group chats are unaffected — the Megolm per-message signature is load-bearing for cross-member anti-impersonation (INVARIANTS §24).
