Where we are right now

  • Phases 1 → 5 are implemented and test-green. Transport, E2EE, at-rest hardening, obfuscation, DHT mailbox, sealed sender, and Megolm-style group chats are all on disk with unit-test and audit-pack coverage. cargo test --lib: 119 / 119.
  • F12 (self-audit) actioned — outbox.peer_id no longer plaintext at rest. commit 74328e7
    The persistent outbox stored the recipient PeerId in the clear, so a disk-read attacker without the OS-keyring DEK could enumerate who had mail queued even though the message body was encrypted. The column now holds HMAC-SHA256(DEK, "zerocenter-outbox-peer-v1" || peer_id) — deterministic, so the by-peer lookup and idx_outbox_peer index are unchanged; one-way, which is enough since the drain path always already holds the PeerId. A PRAGMA user_version gate re-tags pre-F12 rows once on first open.
  • OTPK serve/consume regression — found in live DM testing, fixed 2026-05-20 commit b365be4
    The audit-F3 fix (835c299) gated load_otpk_private on consumed_at IS NULL, but pop_unused_otpk sets consumed_at at serve time — so the private OTPK was refused for the very first legitimate responder X3DH, and every 3-DH first message was dropped with OTPK id=N not found in our store, permanently desyncing the session. The lifecycle is now split across two columns: served_at (set on pop, blocks re-serving) and consumed_at (set after a successful first-decrypt — the F3 replay gate). Migrates pre-fix DBs via ALTER TABLE + backfill. The F3 unit test that encoded the bug was replaced with a full serve → load → consume lifecycle test.
  • Codebase has not had an external audit. Phase 3-5 implement known-good algorithms (Signal Double Ratchet, X3DH-lite, Megolm-style sender chains, vetted primitives) but the integration is hand-rolled. Treat as research-grade until third-party review.
