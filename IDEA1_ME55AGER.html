Vision

ME55 is a censorship-resistant, leaderless, zero-trust P2P messenger. The design constraints are stronger than Signal's because there is no central server, no operator, and no bootstrap-must-be-online dependency beyond what the user explicitly configures with -B/--bootstrap. Identity is Ed25519. Direct messages are Signal-spec Double-Ratchet encrypted on top of an X3DH-lite handshake. At-rest data is AEAD-encrypted with a per-profile key from the OS keyring. Optional ChaCha20 wire obfuscation defeats pattern-matching DPI; size-fingerprint hiding is in flight.

The product is a single Rust binary ME55.exe; --profile alice lets multiple instances coexist on one host (separate identity, separate SQLite store, separate keyring entry).

Wire protocols

Direct-message envelope (/ME55/direct-message/2.0.0)

ProtocolMessage (JSON-serialised, cbor-framed by libp2p request-response):
  to:        Vec<u8>          (recipient PeerId bytes)
  from:      Vec<u8>          (sender PeerId bytes)
  payload:   Vec<u8>          (serialised EncryptedPayload)
  timestamp: i64
  ttl:       i64               (default 7 days)
  msg_type:  MessageType       (Direct, Group, File, Voice, Control)
  signature: Vec<u8>          (Ed25519 over canonical signing-bytes)

Signing bytes (NOT JSON — deterministic byte layout):
  b"ME55-dm-v1" || len_be(to) || to || len_be(from) || from
                       || len_be(payload) || payload
                       || be(timestamp) || be(ttl) || msg_type as u8

EncryptedPayload (inside `payload`, also JSON-serialised):
  dh:        [u8; 32]          (sender's current DH ratchet pubkey)
  pn:        u32               (length of previous sending chain)
  n:         u32               (this msg's index in sending chain)
  ct:        Vec<u8>          (ChaCha20-Poly1305 ciphertext + tag)
  x3dh_eph:  Option<[u8;32]>  (initiator's X3DH ephemeral pubkey,
                                only on first message of fresh session)
  otpk_id:   Option<i64>      (responder's local row id of the OTPK
                                the initiator consumed; only on first
                                message when 3-DH was used)

Prekey fetch protocol (/ME55/prekey/1.0.0)

PrekeyRequest:   ()           (no payload — responder knows what's its)
PrekeyResponse:
  x25519_public: [u8; 32]      (responder's signed prekey)
  signature:     [u8; 64]      (Ed25519 over ME55-prekey-v1 || x25519_public)
  otpk:          Option<OneTimePrekey>

OneTimePrekey:
  id:            i64           (responder's local row id; initiator echoes
                                back so responder can look up private bytes)
  x25519_public: [u8; 32]
  signature:     [u8; 64]      (same Ed25519 signing-bytes layout)

Wire obfuscation (when --obfs-key set)

Connection open (in the clear, 12 bytes):
  [12 random bytes : ChaCha20 nonce, dialer-generated]

After handshake, every subsequent byte is XOR'd with ChaCha20(obfs_key, nonce).
Phase 4c.2 framing adds (still XOR'd over the whole frame):
  [u16-be: actual_len] [actual_len bytes payload] [pad to 256-multiple]

The receiver descrambles the 2-byte header, derives payload+pad lengths,
descrambles payload+pad, delivers only payload to upper layer.

Message routing flow — sending a DM

user types `send <peer_id> hello`
        │
        v
cli::run_cli_with_handlers → CommandHandler "send"
        │ sends NodeCommand::Send(peer, "hello") via mpsc
        v
P2PNode::run_with_commands  loop
        │
        v
try_send_or_queue(peer, plaintext)
   │
   ├── if peer not in connected_peers:
   │      store.outbox_add(...) → "📭 queued"; ConnectionEstablished
   │      will retrigger this fn for the same peer later.
   │
   ├── if session exists (in memory or via load_session):
   │      encrypt_and_send_existing(peer, plaintext, hello=None)
   │
   ├── if prekey cached (load_prekey):
   │      bootstrap_initiator_and_send:
   │          ↳ if cached_otpks has bundle: x3dh::initiator_derive_with_otpk → 3-DH
   │          ↳ else:                       x3dh::initiator_derive          → 2-DH
   │          RatchetState::new_initiator(sk, their_prekey_pub)
   │          encrypt_and_send_existing(peer, plaintext, hello=Some{eph, otpk_id?})
   │
   └── else:
          pending_sends[peer].push(plaintext)
          if not inflight: prekey::send_request(peer, PrekeyRequest)

encrypt_and_send_existing:
   1. ad = ratchet_ad(me, peer)   = len_be(me_pid)||me_pid||len_be(peer_pid)||peer_pid
   2. session.encrypt(plaintext, ad) → RatchetMessage{Header, ciphertext}
   3. payload = EncryptedPayload{dh, pn, n, ct, x3dh_eph?, otpk_id?}
   4. proto = ProtocolMessage::new_direct_signed(peer, me, payload, keypair)
   5. request_response.send_request(peer, DirectMessageRequest(wire_bytes))
   6. store.store_message(...) — local plaintext copy for history
   7. persist_session(peer) — save updated ratchet state, AEAD-encrypted at rest

request_response::Event::Message{Request} fires
        │
        v
process_incoming_dm(transport_peer, request_bytes)
   │
   ├── 1. parse ProtocolMessage from wire bytes
   │
   ├── 2. verify Ed25519 signature → verified_sender PeerId
   │      (the verifying key is extracted from the PeerId multihash;
   │       this works because libp2p Ed25519 PeerIds embed the pubkey

   │      (load-bearing — without this an authenticated peer could relay
   │       a captured message and we'd treat it as direct delivery)

   │
   └── 5. route by session state:
          │
          ├── session exists or restore_session_if_persisted:
          │     decrypt_and_store(sender, payload)
          │       ↳ session.decrypt(ratchet_msg, ad)
          │       ↳ store.store_message + persist_session
          │       ↳ println!("🔓 ...")   user-visible
          │
          └── no session AND payload.x3dh_eph is Some:

Active landmines (read before continuing)

Phase 4c.2 framing has a deadlocking test. short_inner_writes_dont_desync_keystream spins forever because the writer's poll_flush waits for all pending bytes (which includes 22 pad bytes of the frame) to drain into the inner duplex, but the reader's read_exact(1000) returns as soon as the 1000 payload bytes are delivered — it never asks for the pad bytes that follow. The reader's frame state is left at InBody{payload_remaining: 0, pad_remaining: 22} with no caller to drive further poll_read, so the writer waits forever for its pad to be consumed.

Fix options:

  • Change the test to drain the reader to EOF (read_to_end instead of read_exact), and have the writer close to signal EOF after flushing. The pad gets consumed during the drain, and the test asserts on the first 1000 bytes.
  • OR have poll_read eagerly attempt pad consumption right after delivering the last payload byte (non-blocking — return Ok(n) regardless of whether pad drained). This makes pad consumption opportunistic but doesn't fully solve the problem if no further poll_read happens.
  • OR rethink the writer's flush semantics so pad-bytes-in-pending don't block flush (would need a way to distinguish "caller-payload bytes" from "internal padding bytes" in pending).

The first option is the cleanest test fix and matches realistic production usage (libp2p Noise XX upgrade is always reading + writing, so pad gets consumed in the normal flow).

QUIC re-enable is unverified. ring 0.17.14 compiled cleanly on the current MSVC toolchain this session, but we built libp2p WITHOUT the quic feature. Don't claim QUIC works until cargo build with quic in the feature list succeeds AND the swarm-builder gets a .with_quic() call back.
Per-profile DEK is in Windows Credential Manager (generic credential). cmdkey /list doesn't show generic credentials — you need PowerShell + CredEnumerateW via P/Invoke, or the Control Panel GUI. When cleaning up test profiles, the directory under %LOCALAPPDATA%\ME55\<profile> can be rm -rf'd but the keyring entry will linger unless deleted via the GUI. It's a 32-byte random DEK so leaving it is harmless.
Two-peer smoke testing on one host: the listener (bob) must be backgrounded with a kept-open stdin or his CLI loop EOFs and triggers node_handle.abort() before interface listeners can bind. Use sleep N | binary & rather than just binary &. Tear down with pkill -P $! (the whole process group).

Conventions a contributor should know

  • Commit messages name the phase: feat(obfs): Phase 4b — .... Body lists files touched and surface verification.
  • Wire-format changes need both peers on the same version. Bump the /ME55/<protocol>/<ver> string in src/network/behaviour.rs if the framed payload layout changes.
  • Any new info! / warn! that touches decrypted bytes defaults to debug! (INVARIANTS §19). The println!("🔓 ...") lines are deliberate UI, out of scope.
  • audit/INVARIANTS.md is load-bearing. Add invariants with the next number; fixed ones get a "(fixed)" annotation rather than deletion so reviewers see the history of claims.
  • plans/ROADMAP.md is the running roadmap. Refresh it whenever a phase lands.
  • ConnectedPoint for transport .and_then match lives at libp2p::core::ConnectedPoint — the libp2p::core::transport::ConnectedPoint path is private despite being where it's defined (known libp2p 0.53 quirk).
  • The libp2p transport is custom (with_other_transport, not with_tcp). Don't accidentally revert to with_tcp when adding a feature; keep the MaybeScrambled enum as the Output type.
  • SQLite INSERT helpers must return conn.last_insert_rowid(), NOT conn.execute(...)? as i64. The latter returns affected-rows (always 1) and led to a real bug fixed in commit 7caa646.
