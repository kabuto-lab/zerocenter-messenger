ZeroCenter Messenger.

A censorship-resistant, zero-trust, leaderless peer-to-peer communication platform. No servers. No accounts. No central authority. Cryptographic identity, direct peer connections, and local-first storage.

Concept

ZeroCenter is a messenger built on a simple thesis: communication infrastructure should not require trust in any third party. There is no central server, no phone number registry, no account provider. Each participant is cryptographically addressed by the hash of their own Ed25519 public key, and messages flow directly between peers over an encrypted transport.

Zero-Trust

No registration authority, no identity provider. Your keys are your identity; they never leave the device.

Leaderless

No master node, no rendezvous server. Peer discovery via Kademlia DHT and mDNS on local networks.

Censorship-Resistant

Target: pluggable obfuscated transports (Obfs4, Snowflake) so the protocol survives DPI and blocklists.

Local-First

Messages and contacts stored in a per-profile SQLite database. Works offline; syncs when peers come online.

Protocol Flow

  1. Bootstrap: On startup, load or generate an Ed25519 identity per profile.
  2. Listen: Open TCP + QUIC on a random port; advertise via mDNS.
  3. Discover: Peers on the LAN are found via mDNS; global peers via Kademlia DHT (once bootstrap nodes exist).
  4. Connect: Noise Protocol (ChaCha20-Poly1305) handshake, then Yamux stream multiplexing.
  5. Exchange: Direct DMs via request-response over CBOR; offline fallback via Gossipsub topic /zerocenter/global.
  6. Persist: Received and sent messages recorded in the per-profile SQLite database.

Critical Issues to Address

Security gap. Despite the "zero-trust" framing, messages are currently sent in cleartext at the application layer. Noise only protects transport hops, not end-to-end. The SQLite column is named ciphertext but stores content.as_bytes().

Security

  • CRITICAL — No E2EE. x25519-dalek is a declared dependency but unused. Need Double Ratchet or equivalent.
  • CRITICAL — Gossipsub fallback broadcasts DMs globally on /zerocenter/global. Any subscribed peer reads them. This fallback must be removed or replaced with an encrypted-envelope relay scheme.
  • HIGH — No sender signature verification on incoming ProtocolMessage. Sender fields are attacker-controlled.
  • HIGH — No replay protection, no per-message nonces, no forward secrecy.
  • MED — Wire format is serde_json; should be CBOR (already a libp2p feature) or bincode for size and stricter parsing.

Correctness Bugs

  • mDNS discovery pre-populates connected_peers before any connection is established — peers command lies.
  • contacts command ignores the contacts SQLite table and just lists connected peers.
  • Kademlia bootstrap list is empty — DHT cannot be used beyond the LAN.
  • chrono_format computes time-of-day incorrectly and shows no date.
  • Dead code: P2PNode::send_message is unreachable after run_with_commands consumes self.
  • Docs drift: DEV_CONTEXT.md says Phase 1, README.md says Phase 2.

Code Hygiene

  • No tests anywhere — no tests/ directory, no #[cfg(test)] modules.
  • components/ directory is empty.
  • gui module is gated on a gui feature that isn't declared in Cargo.toml.
  • Unused imports / bindings (peer, request_id) will emit warnings on strict builds.

Quick Start

Two instances on the same machine:

cd C:\__Qwen1\ME55
cargo build --release

# Terminal 1
set RUST_LOG=info
target\release\zerocenter.exe --profile alice

# Terminal 2
set RUST_LOG=info
target\release\zerocenter.exe --profile bob

In each REPL: help, peers, connect <multiaddr>, send <peer_id> <msg>, history, quit.
